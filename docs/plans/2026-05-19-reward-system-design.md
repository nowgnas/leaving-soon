# 곧 나가요 리워드 시스템 설계

## 목표

`곧 나가요` 제보를 더 많이 만들기 위해, 현금성 보상이나 외부 쿠폰 없이 서비스 안에서만 작동하는 리워드 구조를 설계한다.

리워드는 사용자가 "내가 제보하면 나도 서비스 안에서 더 좋은 경험을 얻는다"고 느끼게 해야 한다. 동시에 허위 제보, 위치 정보 오남용, 구현 복잡도는 낮게 유지해야 한다.

## 제약 조건

- 추가 결제, 현금, 포인트 환전, 쿠폰, 외부 제휴 보상은 사용하지 않는다.
- 리워드는 서비스 내부 경험으로만 제공한다.
- 현재 MVP에는 로그인과 평판 시스템이 없다.
- 제보는 시간 민감도가 높으므로 최근성, 유용성, 위치 신뢰도를 중심으로 평가한다.
- 위치 정보는 "이 사람이 실제로 카페 근처에 있었는지"를 확인하는 데만 사용한다.
- 정확한 위치 좌표를 장기 저장하지 않는다.

## 추천 방향

서비스 안에서 닫힌 리워드 루프를 만든다.

1. 사용자가 카페 근처에서 `곧 나가요`를 등록한다.
2. 위치가 카페와 가까우면 해당 제보를 `위치 확인 제보`로 표시한다.
3. 사용자는 기여 점수를 받는다.
4. 기여 점수는 서비스 내부 혜택을 연다.
   - 더 명확한 방문자 요약
   - 제보 연속 기록
   - 신뢰 제보자 표시
   - 검증된 제보 기반의 상세 정보
5. 점수와 상태는 시간이 지나며 약해지게 만들어 계속 제보할 이유를 남긴다.

이 방식은 돈처럼 쓰이는 보상을 만들지 않으면서, 서비스의 핵심 가치인 "실시간에 가까운 자리 정보"와 보상을 직접 연결한다.

## 리워드 원칙

- 제보 개수보다 유용성을 보상한다.
- 위치 확인된 최근 제보를 더 높게 평가한다.
- 기본 방문자 기능을 막는 paywall처럼 만들지 않는다.
- 제보자의 정확한 위치를 노출하지 않는다.
- 스팸 제보가 이득이 되는 구조를 만들지 않는다.
- 첫 버전은 로그인 없이도 작동해야 한다.

## 리워드 종류

### 1. 기여 점수

유효한 제보를 남기면 기여 점수를 받는다.

추천 점수:

| 행동 | 점수 |
|---|---:|
| 유효한 제보 등록 | +1 |
| 카페 근처 위치 확인 성공 | +2 |
| 콘센트, 웨이팅, 자리 넓이 등 상세 정보 포함 | +1 |
| 방문자가 `도움 됐어요` 피드백 | +2 |
| 중복 또는 의심 제보 | 0 |

기여 점수는 돈이 아니며 양도하거나 환전할 수 없다. 서비스 내부 상태와 정보 경험에만 영향을 준다.

### 2. 제보 연속 기록

반복 기여를 가볍게 보상한다.

예시:

- `오늘의 자리 제보자`: 오늘 위치 확인 제보 1건 이상
- `3일 연속 도움`: 서로 다른 날짜에 위치 확인 제보 3일
- `성수 도우미`: 같은 지역에서 반복적으로 검증된 제보

연속 기록은 너무 엄격하면 부담이 된다. 하루 놓쳤다고 사용자가 손해를 크게 느끼지 않게 설계한다.

### 3. 신뢰 제보자 표시

위치 확인 제보가 누적되면 제보 신뢰도를 표시한다.

추천 단계:

- `새 제보자`: 기본 상태
- `확인된 제보자`: 위치 확인 제보 3건
- `믿을 만한 제보자`: 위치 확인 제보 10건 이상이고 잘못된 제보 비율이 낮음

이 표시는 개인을 과하게 드러내는 평판이 아니라, 제보 품질에 대한 신뢰 신호로 사용한다. 방문자 화면에서는 `위치 확인 제보`, `확인된 제보`처럼 익명 상태로 표시하는 것이 좋다.

### 4. 방문자 정보 혜택

기여자는 방문자 화면에서 조금 더 풍부한 정보를 볼 수 있다.

기본 자리 정보는 모든 사용자에게 열어둔다. 대신 기여자에게는 다음 정보를 더 명확히 보여준다.

- 더 구체적인 요약 문장
- 제보 최근성 구간
- `최근 몇 분 내 제보 기준` 강조
- `검증된 제보 중심` 같은 신뢰도 라벨
- 자주 보는 카페의 간단한 흐름 정보

핵심 기능을 잠그지 않고, 제보한 사용자에게 "조금 더 좋은 정보 경험"을 제공하는 방식이다.

### 5. 고마움 피드백

방문자는 제보를 보고 간단한 피드백을 남길 수 있다.

예시:

- `도움 됐어요`
- `이미 자리 없음`
- `정보가 달라요`

좋은 피드백은 제보자 점수에 반영한다. 잘못된 피드백이 반복되는 제보자는 신뢰도를 낮춘다. 피드백은 익명 1-tap으로 유지한다.

## 위치 기반 검증

리워드가 생기면 위치 기반 검증이 필요하다. 사용자가 실제 카페 근처에 있지 않아도 점수를 받을 수 있으면 제보 품질이 빠르게 무너진다.

### 클라이언트 흐름

1. 사용자가 카페를 선택한다.
2. `곧 나가요` 입력 화면을 연다.
3. 등록 시 위치 권한을 요청한다.
4. 안내 문구:
   - `이 카페 근처에서 남긴 제보인지 확인해요. 정확한 위치는 저장하지 않습니다.`
5. 브라우저 geolocation으로 위도, 경도, 정확도를 받는다.
6. 클라이언트가 서버에 함께 보낸다.
   - 선택한 카페 ID
   - Naver 검색 결과의 카페 좌표
   - 사용자 현재 좌표
   - 위치 정확도
   - 위치 수집 시각
7. 서버가 위치 검증 결과를 계산한다.

### 서버 검증 정책

서버는 사용자 위치와 카페 좌표 사이의 거리를 계산한다.

추천 기준:

| 조건 | 결과 |
|---|---|
| 거리 150m 이하, 위치 정확도 100m 이하 | `verified` |
| 거리 300m 이하, 위치 정확도 200m 이하 | `weak_verified` |
| 위치 권한 거부 | `unverified` |
| 거리 300m 초과 | `unverified` |

추가 리워드는 `verified`, `weak_verified`에만 준다. 단, 위치 권한을 거부해도 제보 자체는 가능하게 둔다.

### 개인정보 원칙

- 사용자 원본 위도/경도는 장기 저장하지 않는다.
- 저장할 값:
  - `location_verification_status`
  - 거리 구간: `0-150m`, `150-300m`, `300m+`
  - 검증 시각
- 운영 로그에 원본 좌표를 남기지 않는다.
- 위치 권한은 선택 사항으로 둔다.
- 위치 확인 없이 등록된 제보는 표시하되, 신뢰도와 리워드는 낮게 준다.

## 사용자 식별 방식

리워드를 주려면 기여자를 어느 정도 식별해야 한다.

### 1단계: 익명 기기 ID

브라우저 localStorage에 랜덤 contributor ID를 생성한다.

장점:

- 로그인 없이 바로 쓸 수 있다.
- 구현이 빠르다.
- 현재 MVP 구조와 잘 맞는다.

단점:

- 브라우저 저장소를 지우면 기여 기록이 사라진다.
- 기기를 바꾸면 이어지지 않는다.
- 계정보다 악용 방지가 약하다.

첫 버전은 이 방식을 추천한다.

### 이후: 선택형 계정

사용자가 원하면 이메일이나 소셜 로그인으로 기여 기록을 저장하게 한다.

장점:

- 기여 기록을 여러 기기에서 유지할 수 있다.
- 신뢰 제보자 관리가 쉬워진다.
- 악용 대응이 쉬워진다.

단점:

- 인증, 개인정보 처리, 계정 관리가 추가된다.
- 초기 제보 전환율을 낮출 수 있다.

리워드 루프가 실제로 작동하는 것을 확인한 뒤 도입한다.

## 악용 방지

기본 방어:

- 같은 카페에서 같은 사용자가 받을 수 있는 리워드 횟수를 제한한다.
- 예: 카페별 1시간에 최대 2건만 리워드 인정.
- 같은 사용자의 유사 제보를 중복 처리한다.
- 방문자 피드백에서 `정보가 달라요`가 반복되면 신뢰도를 낮춘다.
- 신뢰 제보자 단계는 위치 확인 제보만 반영한다.
- 위치 미확인 제보는 노출하되 낮은 신뢰도로 표시한다.

위치 권한이 없다고 제보를 완전히 막지는 않는다. 실내에서는 GPS가 흔들릴 수 있고, 위치 권한을 거부하는 사용자도 있기 때문이다.

## UX 흐름

### 제보자

1. 카페를 선택한다.
2. `곧 나가요` 폼을 입력한다.
3. `곧 나가요` 버튼을 누른다.
4. 앱이 위치 권한을 요청한다.
5. 등록이 성공한다.
6. 완료 화면에서 보여준다.
   - `곧 나가요가 등록되었습니다.`
   - `위치 확인 제보로 신뢰도가 올라갔어요.`
   - `오늘의 기여: 3점`

위치 권한을 거부한 경우:

- `위치 확인 없이 등록되었어요. 다음에는 위치 확인을 켜면 더 높은 신뢰도를 받을 수 있어요.`

### 방문자

방문자 요약에는 다음을 표시할 수 있다.

- `검증된 제보 2건 포함`
- `최근 12분 내 제보 기준`
- 제보 row에 `위치 확인` 배지

제보를 본 뒤에는 피드백을 받을 수 있다.

- `이 제보가 도움이 됐나요?`
- 버튼: `도움 됐어요`, `달랐어요`

## 단계별 출시 계획

### Phase 1: 소프트 리워드 MVP

- 익명 contributor ID 생성
- 제보 등록 시 위치 확인
- 등록 후 리워드 메시지 표시
- 위치 검증 상태와 지급 점수 저장
- 방문자 화면에 위치 확인 배지 표시

### Phase 2: 신뢰도와 피드백

- 방문자 피드백 추가
- 기여 점수 합계 추가
- 신뢰 제보자 단계 추가
- 기본 악용 방지 추가

### Phase 3: 선택형 계정

- 기여 기록을 계정에 저장
- 내 기여 상태 화면 추가
- 필요할 경우 지역 기반 배지 추가

## 지표

추적할 지표:

- 제보 등록 전환율
- 위치 권한 허용률
- 위치 확인 제보 비율
- 방문자 1명당 제보 수
- 반복 제보자 비율
- `도움 됐어요` 피드백 비율
- `정보가 달라요` 피드백 비율
- 사용자/카페별 의심 제보 비율

## 위험과 대응

### 위험

- 위치 권한 요청이 제보 전환율을 낮출 수 있다.
- 단순 점수 보상이 스팸 제보를 만들 수 있다.
- 익명 기기 ID는 초기화될 수 있다.
- 과한 게임화는 서비스 신뢰도를 낮출 수 있다.

### 대응

- 위치 권한은 선택으로 둔다.
- 제보 개수보다 위치 확인과 방문자 피드백을 더 크게 보상한다.
- 배지는 과장하지 않고 사실 기반으로 표시한다.
- 카페별/사용자별 리워드 제한과 중복 감지를 둔다.

## 최종 추천

Phase 1부터 시작한다.

첫 버전에서는 로그인 없이 익명 기기 ID, 선택형 위치 확인, 등록 후 리워드 메시지만 넣는다. 이렇게 하면 현재 MVP 구조를 크게 흔들지 않으면서도 사용자가 `곧 나가요`를 등록할 이유를 만들 수 있다.
